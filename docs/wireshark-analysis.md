# Wireshark DNS Packet Analysis — Lab Guide

## Objective

Capture live DNS traffic with Wireshark, inspect DNS query and response packets at the byte level, and understand every field in the DNS header.

---

## Step-by-Step Lab Procedure

### Step 1 — Start Wireshark

```bash
# Launch with GUI
sudo wireshark &

# Or use tshark (terminal-based)
sudo tshark -i any -f "port 53" -V 2>/dev/null | head -100
```

Select your active network interface (e.g., `eth0`, `wlan0`, `ens33`).

### Step 2 — Generate DNS Traffic

Open a second terminal and run:

```bash
# Standard A record query
dig www.google.com

# This generates two packets:
#  Packet 1: DNS Query   (your machine → resolver)
#  Packet 2: DNS Response (resolver → your machine)
```

### Step 3 — Apply the DNS Filter

In the Wireshark display filter bar, type:
```
dns
```
Press Enter. Only DNS packets are now visible.

---

## Analysing the Query Packet

**Double-click the first packet** (the query — Source = your IP).

Expand `Domain Name System (query)` in the packet details pane.

### Full Field Breakdown

```
Domain Name System (query)
│
├── Transaction ID: 0x1a2b
│   ├── Size: 16 bits (2 bytes)
│   ├── Random value generated by the client
│   └── The response must echo the same ID → this is how client matches
│       responses to queries. Kaminsky attack targets this field.
│
├── Flags: 0x0100
│   ├── Total size: 16 bits
│   │
│   ├── QR (Query/Response): 0
│   │   └── 0 = This is a QUERY
│   │       1 = This is a RESPONSE
│   │
│   ├── Opcode: 0000 (Standard Query)
│   │   ├── 0 = QUERY  (standard lookup)
│   │   ├── 1 = IQUERY (inverse query — deprecated)
│   │   └── 2 = STATUS (server status request)
│   │
│   ├── AA (Authoritative Answer): 0
│   │   └── Not relevant in queries; set in responses from auth servers
│   │
│   ├── TC (Truncated): 0
│   │   └── 0 = Message is complete
│   │       1 = Message was cut off (retry over TCP)
│   │
│   ├── RD (Recursion Desired): 1
│   │   └── 1 = Client wants the resolver to find the answer recursively
│   │       0 = Client only wants what the server already knows
│   │
│   ├── RA (Recursion Available): 0
│   │   └── Set by the server in responses; not relevant in queries
│   │
│   ├── Z (Reserved): 0
│   │   └── Must always be 0
│   │
│   ├── AD (Authentic Data): 0
│   │   └── Set by DNSSEC-validating resolvers in responses
│   │
│   └── CD (Checking Disabled): 0
│       └── 1 = Client accepts non-validated DNSSEC data
│
├── Questions: 1
│   └── Number of entries in the Question section
│
├── Answer RRs: 0
│   └── 0 in queries; number of answers in responses
│
├── Authority RRs: 0
│   └── 0 in most queries
│
├── Additional RRs: 1
│   └── Usually 1 for the OPT (EDNS0) pseudo-record
│
└── Queries
    ├── Name: www.google.com
    │   └── Domain being looked up (encoded in DNS label format)
    │
    ├── Type: A (1)
    │   ├── 1  = A (IPv4 address)
    │   ├── 28 = AAAA (IPv6)
    │   ├── 15 = MX
    │   ├── 2  = NS
    │   ├── 5  = CNAME
    │   ├── 6  = SOA
    │   ├── 16 = TXT
    │   ├── 12 = PTR
    │   └── 252= AXFR (zone transfer)
    │
    └── Class: IN (1)
        └── IN = Internet (always 1 for standard DNS)
```

## Analysing the Response Packet

**Double-click the second packet** (the response — Source = resolver IP).

```
Domain Name System (response)
│
├── Transaction ID: 0x1a2b
│   └── MUST match the query Transaction ID
│
├── Flags: 0x8180
│   ├── QR: 1                    ← This is a RESPONSE
│   ├── Opcode: 0                ← Standard query
│   ├── AA: 0                    ← 1 if from authoritative server directly
│   ├── TC: 0                    ← 1 if response was truncated (>512 bytes)
│   ├── RD: 1                    ← Echoed from the query
│   ├── RA: 1                    ← This server DOES support recursion
│   ├── Z: 0
│   ├── AD: 0                    ← 1 if DNSSEC validated
│   ├── CD: 0
│   └── RCODE: 0 (No error)      ← 3=NXDOMAIN, 2=SERVFAIL, 5=REFUSED
│
├── Questions: 1
├── Answer RRs: 6                 ← 6 IP addresses returned
├── Authority RRs: 0
├── Additional RRs: 1
│
├── Queries
│   └── (echoed from the original query)
│
└── Answers
    ├── Resource Record 1
    │   ├── Name:        www.google.com
    │   ├── Type:        A (1)
    │   ├── Class:       IN (1)
    │   ├── TTL:         300        ← Cache this for 300 seconds (5 min)
    │   ├── Data length: 4          ← IPv4 = 4 bytes
    │   └── Address:     142.250.190.4
    │
    ├── Resource Record 2
    │   └── Address:     142.250.190.68
    │
    └── ... (more A records — Google has many)
```

---

## RCODE Values Explained

| RCODE | Name | Meaning | When You See It |
|-------|------|---------|----------------|
| 0 | NOERROR | Success | Normal resolution |
| 1 | FORMERR | Format error | Malformed query |
| 2 | SERVFAIL | Server failure | DNSSEC validation failed; resolver error |
| 3 | NXDOMAIN | Non-existent domain | Domain does not exist |
| 4 | NOTIMP | Not implemented | Server doesn't support this query type |
| 5 | REFUSED | Refused | Zone transfer denied; policy rejection |

## Practical Wireshark Exercises

### Exercise 1: Standard A Query

```bash
dig www.google.com
```

In Wireshark, observe:
- UDP packet on port 53 (query)
- UDP response with one or more A records
- Transaction IDs match between query and response
- RD=1 in query, RA=1 in response

---

### Exercise 2: NXDOMAIN — Non-Existent Domain

```bash
dig notarealadomain999xyz.com
```

In Wireshark:
- Apply filter: `dns.flags.rcode == 3`
- Observe: RCODE = 3 (NXDOMAIN) in response flags
- Answer section: empty
- Authority section: SOA record of the TLD (negative caching)

---

### Exercise 3: Zone Transfer (TCP)

```bash
dig axfr @nsztm1.digi.ninja zonetransfer.me
```

In Wireshark:
- Apply filter: `tcp.port == 53`
- Observe: TCP 3-way handshake (SYN → SYN-ACK → ACK)
- Multiple DNS response packets (zone data is large)
- TCP FIN when transfer completes

---

### Exercise 4: Forced TCP Query

```bash
dig +tcp @8.8.8.8 google.com
```

In Wireshark:
- Apply filter: `tcp.port == 53`
- Observe: TCP handshake before DNS data appears
- Compare with UDP query: no handshake

---

### Exercise 5: DNSSEC Query

```bash
dig @1.1.1.1 cloudflare.com A +dnssec
```

In Wireshark:
- In the query, expand the OPT record (Additional section)
- Observe: DO bit = 1 (DNSSEC OK — client wants signatures)
- In the response, observe RRSIG records alongside A records
- Check flags for AD = 1 (Authenticated Data)

## DNS Packet Filter Summary

| What to Observe | Wireshark Filter |
|----------------|-----------------|
| All DNS traffic | `dns` |
| Queries only | `dns.flags.response == 0` |
| Responses only | `dns.flags.response == 1` |
| NXDOMAIN | `dns.flags.rcode == 3` |
| SERVFAIL | `dns.flags.rcode == 2` |
| Authoritative answers | `dns.flags.authoritative == 1` |
| DNSSEC validated (AD) | `dns.flags.authenticated == 1` |
| Truncated (TC) | `dns.flags.truncated == 1` |
| Specific domain | `dns.qry.name == "google.com"` |
| A record queries | `dns.qry.type == 1` |
| MX queries | `dns.qry.type == 15` |
| AXFR requests | `dns.qry.type == 252` |
| DNS over TCP only | `tcp.port == 53` |
| DNS over UDP only | `udp.port == 53` |
| From specific server | `dns && ip.src == 8.8.8.8` |

## DNS Label Encoding

Domain names are stored in DNS packets using **length-prefixed labels**:

```
www.google.com  →  03 77 77 77 06 67 6f 6f 67 6c 65 03 63 6f 6d 00
                    ↑  w  w  w  ↑  g  o  o  g  l  e  ↑  c  o  m  ↑
                    3 chars     6 chars                3 chars     null terminator

Each label is preceded by a 1-byte length.
The sequence ends with a 0x00 null byte.
```

This encoding is visible in the raw packet bytes pane in Wireshark (bottom panel).
